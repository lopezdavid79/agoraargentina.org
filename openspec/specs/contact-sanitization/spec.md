# Contact Sanitization Specification

## Purpose

Define how the system sanitizes user input submitted through the public contact form and ensures all dynamic output rendered in EJS templates is HTML-escaped to prevent XSS attacks.

## Requirements

### Requirement: Input Sanitization

The system MUST sanitize every user-supplied field in the contact form (`nombre`, `email`, `telefono`, `asunto`, `mensaje`) before the data is used in email composition or stored.

HTML tags and script fragments injected by a user MUST be stripped or neutralized so they are not interpreted as markup in the email body or in any downstream rendering context.

#### Scenario: Valid contact submission

- GIVEN a user fills the contact form with plain text values
- WHEN the form is submitted
- THEN the email is sent with the exact text values preserved
- AND the user sees a success confirmation

#### Scenario: XSS payload in name and message

- GIVEN a user submits `<script>alert('xss')</script>Juan` as `nombre`
- AND `<p>Hola <strong>mundo</strong></p>` as `mensaje`
- WHEN the form is processed
- THEN the outgoing email MUST NOT contain `<script>`, `<p>`, or `<strong>` tags
- AND the raw text content (`alert('xss')Juan`, `Hola mundo`) MUST still be present

#### Scenario: HTML in phone field

- GIVEN a user submits `<b>123456789</b>` as `telefono`
- WHEN the form is processed
- THEN the outgoing email MUST NOT contain `<b>` tags
- AND the numeric text `123456789` MUST be preserved

#### Scenario: Empty optional fields

- GIVEN a user leaves `telefono` and `mensaje` empty
- WHEN the form is submitted
- THEN the email is still sent
- AND empty fields are rendered as empty or with a default placeholder

### Requirement: Output Escaping in EJS Templates

The system MUST render all dynamic user-controlled values in EJS templates using the escaped output tag `<%= ... %>` instead of the unescaped tag `<%- ... %>`.

Any template that currently outputs raw HTML from user data via `<%-` MUST be updated to use `<%=` unless the rendered content is explicitly trusted markup generated by the system itself.

#### Scenario: News detail page renders escaped content

- GIVEN a news article contains a `<script>` payload in its `contenido`
- WHEN a visitor opens the news detail view
- THEN the script tag is displayed as literal text
- AND the browser MUST NOT execute the injected script

#### Scenario: Admin news detail renders escaped content

- GIVEN an admin user opens the detail page for a news article with HTML in the body
- WHEN the page is rendered
- THEN the HTML is escaped and shown as text rather than interpreted by the browser

#### Scenario: Course detail page renders escaped content

- GIVEN a course description contains an `<img onerror=...>` payload
- WHEN a visitor opens the course detail view
- THEN the payload is escaped and shown as text

#### Scenario: Training detail page renders escaped content

- GIVEN a training module description contains an `<iframe>` payload
- WHEN a visitor opens the training detail view
- THEN the iframe tag is escaped and shown as text

### Requirement: Sanitization Does Not Break Email Delivery

The system MUST NOT reject or corrupt a contact submission solely because it contains special characters, as long as those characters are sanitized to safe text.

#### Scenario: Unicode and special characters preserved

- GIVEN a user submits `María` with an accent and `mensaje` with emojis
- WHEN the form is processed
- THEN the email contains `María` and the emojis intact
- AND no HTML tags are introduced
